Melodick 总体设计说明书：MVP阶段

# 产品定位与核心理念

## AI 重合成赋能的降维打击
> 都是同龄人，我原本没想降维打击！
Melodick 是一款基于“AI重合成”而非传统“机械DSP处理”的人声编辑软件。传统的修音软件（如 Melodyne）基于相位声码器（PSOLA等）对音频进行机械的变速变调，不可避免地引入电音、共振峰撕裂和伪影。
Melodick 的核心杀手锏是：用户在界面上进行的任何音高、时值操作，最终都会转化为基础特征模型（F0 + Mel/时长映射），交由后端的 **NSF-HiFiGAN 引擎** 重新“唱”出来。这是从物理级修改到生成级再造的降维打击。

## 专注人声单旋律的精准定位
Melodick 严守 **Good in，Good out** 原则：
- 目标对象仅为**单旋律干声人声**。
- 不做复音和弦分离，不面向器乐修音。
- 明确不包含去混响、降噪、人声分离等前处理能力。输入如果是被污染的音频，处理效果概不负责。

## 真信源与派生计算原则（核心基石）
在软件的整个生命周期与内存模型中，必须严格区分“真信源（Source of Truth）”与“派生属性（Computed）”：
- **真信源**：用户的编辑操作参数（整体移调量、时值缩放倍率、HandDraw 数据点、LineControl 控制节点）。
- **派生属性**：最终用来显示的音高曲线（EditedPitchSlice）、试听用的音频流、界面波形。
  **严禁逆向修改或多头维护**。派生属性只允许通过真信源正向计算得出；UI 展示和试听系统只能消费派生属性，绝不可将其作为存储依据。

# 运行形态与会话环境模型

## 环境形态的差异化本质
Melodick 通过 JUCE 框架构建，必须在代码架构层面明确区分 Standalone（独立运行）与 Plugin（VST3/AU 等插件）两种形态。它们的生命周期、会话主导权截然不同。

## Standalone 形态：完整会话（Session）结构
在独立运行模式下，Melodick 是整个工程的绝对主宰，拥有完整的会话结构。

### 会话级全局元数据
会话持有全局走带控制器（Play/Pause/Stop/Locate）。
持有工程的全局时间规约：当前基准采样率（内部固定为 44.1kHz）、节拍拍号（Time Signature）、BPM。MVP 阶段支持固定 BPM 与拍号，但架构上必须对接 DAW 的时间线抽象，预留未来支持可变拍号与 BPM 的能力。

### 多轨道（Tracks）体系
支持多轨道编排。轨道拥有独立的混音属性：
- 轨级音量增益（Gain dB）
- Mute（静音）与 Solo（独奏）状态约束（有 Solo 时屏蔽非 Solo 轨，无 Solo 时播放所有非 Mute 轨）。
- 轨道标识名与顺序索引。
- 轨道作为物理容器，内部严格持有自身的 `NoteBlobs` 集合。

## Plugin（VST）形态：受限的伴生会话
在宿主 DAW 中，Melodick 仅作为一个挂接在单轨上的效果器/乐器插件。

### 单一轨道约束与音量增益
Plugin 实例内部有且**仅有一条轨道**。废弃 Mute/Solo 控制逻辑（由宿主接管），仅保留轨级总体音量增益（Gain dB），输出给宿主的 `processBlock`。

### 走带同步与内部覆写逻辑
插件必须与宿主的 Transport 状态严格同步：
- **宿主播放中**：Melodick 的界面强制跟随宿主时间线“播放”，内部走带控制器禁用。当然，“播放”时宿主读的是
- **宿主暂停/停止时**：允许用户在插件 UI 内部执行任意位置的播放、暂停与进度条拖拽（内部走带控制生效），便于微观编辑时的局部试听。

# 音频注入与分析管线（Ingestion Pipeline）

## 标准化工作域转换
为保证 AI 模型（ONNX 推理）的舒适区与减少重复计算，所有进入 Melodick 域的音频，必须在进入的瞬间进行格式归一化。
**内部法定格式永远且仅为：`44.1kHz / Mono / Float32`**。无论用户导入 48k 立体声还是 96k 24-bit 文件，立刻转码，后续的肢解、编辑、渲染一律基于此法定格式。

## 音频注入入口逻辑

### Standalone：文件级导入
用户通过拖拽或菜单导入音频文件到特定轨道的指定时间点。系统记录此初始 Offset 时间，随后立刻触发分析管线。

### Plugin：内录捕获机制（Inner-Record）
由于 VST 插件无法直接主动获取 DAW 的全轨离线文件，采用内录机制：
1. 用户在插件中点击“开启内录模式”。
2. DAW 开始播放，插件在 `processBlock` 中高频捕获输入的音频流。
3. DAW 暂停或用户关闭内录，捕获结束。
   **覆盖与叠加语义**：在已有 NoteBlob 的区间内进行二次录音，会导致**叠加**（新旧重叠存在，预期行为）。若用户逻辑上意图“重录”，必须手动框选删除已有 NoteBlob后再录。

## 分析与肢解流水线
音频一旦注入成功，必须强制经过单向分析流水线。

### 原始 F0 与特征提取（RMVPE 引擎）
音频数据被送入 **RMVPE 模型**。该模块输出高精度的连续特征：原始 F0 数组、发声/清音（Voiced/Unvoiced）状态、置信度、以及能量包络。

### 智能切分与NoteBlob实例化
**这步是产品化最关键的门槛。**
分割器读取 RMVPE 的多模态特征，基于传统混合算法（MVP 阶段暂不引入 AI 分割，以后再议）进行切段。依据：Pitch jump（音高突变）、V/UV 切换、能量短谷等特征，将连续的时间线切割。
**结果输出**：在此阶段，连续的音频流与 F0 流被正式肢解，实例化为一个一个的 `NoteBlob` 对象。至此，全局音频不复存在，NoteBlob 成为唯一操作实体。

# 核心数据模型：NoteBlob（音符块）

## NoteBlob 的资产私有化
`NoteBlob` 是 Melodick 中的“神”。分割完成后，每个 NoteBlob 像一个极度内聚的密封舱，**私有化持有**自己这段时间对应的所有资产：
- `source_audio_44k`：该段源波形片段。
- `original_pitch_curve`：该段的原始 F0 切片。
- `source_f0_hz`：RMVPE 提取出的源帧级 F0 条件。
- `source_voiced_probability`：RMVPE 提取出的源帧级发声概率 / UV 条件。
- `cached_source_mel_log`：该段的运行期梅尔缓存。它是可丢弃的派生缓存，用于加速部分重合成路径，**不属于工程持久化真相**。

## 单向顺序管线状态模型
NoteBlob 内的最终 F0 计算规则必须是一条单向的、不可逆的纯函数（Pure Function）。

### 持久化编辑（真信源）和计算管线顺序
管线的输入即用户的编辑参数：

1. `handdraw_patch`：手绘覆盖层。
2. `line_patches`：结构化曲线控制（颤音、转音/滑音）。
3. `global_pitch_delta_midi`：音高垂直移动产生的偏移。不局限于半音（整数是半音，小数是音分），但在UI层面，如果不按住Alt，那么上下移动默认是以半音为尺度（和Melodyne对齐）。
4. `time_ratio`：左右拉伸产生的时值缩放比。

渲染的输入物料和UI的展示，是按这个管线处理顺序来搞的。

### 运行时动态计算属性
管线按以下公式死板执行：
`EditedPitchSlice = ( Original F0 覆盖 HandDraw + 施加 LineControl影响 ) + Global Transpose，最后按照  进行时间轴重采样拉伸`。
计算得出的 `EditedPitchSlice` 与受时值影响的起止时间交由 UI 显示，并作为 AI 引擎的输入。

### MVP 的前缀清音并入策略（当前实现）
MVP 阶段允许一个 `NoteBlob` 的起点带有一小段前缀清音（例如辅音音头），以避免把真正的发音起始“吞掉”。较长的纯清音段（呼吸、停顿、无声擦音）则保留为独立 Blob，默认直通。
对于较长的纯清音 Blob，分割器会额外做一层边界再分配：
- 最前约 `30ms` 并入左侧紧邻的 voiced Blob；
- 最后约 `60ms` 并入右侧紧邻的 voiced Blob；
- 中间剩余部分继续作为独立 unvoiced Blob 保留。
这意味着当前 `NoteBlob.time` 在实现层更接近“渲染范围（Render Span）”，而不总是严格等于“有音高核心范围（Pitched Core Span）”。

## 音符块拓扑关系网

### 物理重叠与独立渲染叠加语义
这是 Melodick 彻底告别传统修音的一环：**独立的 NoteBlob 允许在时间轴上相互重叠。**
重叠没有任何问题。因为在渲染语义上，每个 NoteBlob 本来就是各自依据自己的 F0 和 Mel 进行独立 AI 渲染，最终在时间轴上**混音相加（Sum）**。

### 逻辑连接（Link）与边界互推
- **相邻音符处于连接状态（Linked）**：左音符向右拉伸时，必然会挤压/推动右音符的左边界。它们之间维持拓扑连续性。
- **解除连接状态（Detached）**：断开后，左右音符完全自由，拉伸不再互推，可以任意移动直至重叠。

### 渲染组（RenderGroup）合并机制
为了兼顾 AI 渲染的丝滑度与计算开销：对于处于 **“连接状态且时间无缝”** 的一系列 NoteBlob，渲染引擎会在底层将其临时合并为一个 `RenderGroup`，把拼接后的合并特征喂给 NSF-HiFiGAN 进行一次长序列生成，避免音符交界处的伪影。

# 懒渲染管线与状态机

## 渲染触发与脏区标记（Dirty Pipeline）
不采用“每动一下就全局重算”的蠢办法。系统依托时间轴实现**懒渲染（Lazy Rendering）**。
只要改变了某个 NoteBlob 的真信源（位置、时值、内部曲线等），UI 线程只需立刻将其时间区间标记为 **“脏（Dirty）”**。在全局状态条上，该时间段瞬间从绿变**红**。

## 异步分块渲染调度
渲染队列独立于音频线程。
- 后台轮询脏区。
- 根据当前播放头的位置（优先渲染即将听到的部分）提取脏区对应的 NoteBlobs 或 RenderGroups。
- 送入 ONNX 推理。
- 推理完成后，写入对应缓存，状态条相应区域复**绿**。

## AI 重合成引擎（NSF-HiFiGAN）

### 输入组装与推理
引擎接收：与当前 RenderGroup 时间轴严格对齐的 Mel/F0/UV 条件。Mel 优先复用单 Blob 的运行期缓存，无法复用时按当前组装后的源音频即时重算；F0 与 UV 则依据 `EditedPitchSlice` 和 RMVPE 提供的发声置信度重新插值到当前 mel 帧网格。输出全新的、完美拟合人声特质的重合成音频（44.1k float32）。

### 原始直通策略（Pass-through）
极端节约算力的黄金法则：如果在检测时发现某 NoteBlob `EditedPitchSlice == OriginalPitchSlice` 且 `time_ratio == 1.0` 且响度未变，引擎直接**旁路跳过推理**，原封不动地返回其持有的 `source_audio_44k` 作为渲染结果。

## 缓存机制（在 UI 试听与 Plugin 宿主输出）
- **走带回放缓存**：基于块级别的离散缓存，满足内部走带播放。
- **宿主专用连续缓存（插件专有）**：因为 VST 宿主会以极高的频率（如 256 samples）死板调用 `processBlock`，我们必须有一个平铺好的环形缓冲（Ring Buffer）供其读取。若宿主播放到了某个“标红（脏）”未渲染完毕的区间，**预期行为就是直接输出静音**，绝不允许阻塞宿主音频线程等待推理。

# 音符块通用编辑语义

## 选中与焦点
单选音符块，右侧边栏展开其专有属性；多选时进入批量控制逻辑（统一平移、统一量化等）。

## 时间轴与空间操作

### 自由拖拽移动与吸附
鼠标直接拖拽 NoteBlob。可选择按毫秒级自由移动，或打开吸附开启（未来按小节/拍子吸附，MVP阶段可暂无此吸附）。

### 边缘拉伸与时值重算
左右拖动音符块边缘改变时值。
**红线规定**：严禁使用任何机械的 Time-Stretch DSP 算法处理音频。拉伸动作的本质，是修改 NoteBlob 的 `time_ratio` 参数，并将其标脏。由后端 AI 引擎基于拉伸后的梅尔与 F0 重新“生成”变长/变短的声音。
*注：目前不支持类似 Melodyne 的“保持总长度不变，在内部切个锚点改变发音长短比”的高级特性，若需此操作，用户必须切分为两个音符块单独拉伸。*

### 切分、合并与连接管理
- **切分（Split）**：将一个 Blob 一分为二，特征与源音频按时间点劈开，新建两个实体。
- **合并（Merge）**：选择相邻的 Blob 进行内部数据拼合。
- **连接管理**：手动打断连线，或强制重新缝合。

## 基础音高与响度控制

### 垂直平移与音级中心量化
上下拖拽不仅改变 `global_pitch_delta_midi`。双击音符，执行吸附：计算当前曲线的核心能量音级，将其中心点精准拍向标准半音网格线上。

### 局部包络与相对响度调节
除了全局增益，每个 NoteBlob 具备基于乘数（Multiplier）的独立响度调节能力，不改变特征结构，仅影响后期合成出波形后的电平控制。

## 历史操作与精准恢复
依托纯函数模型，撤销（Undo/Redo）只操作“真信源”参数栈。
右侧边栏提供精确恢复按钮：恢复原音高（清空全局偏移与曲线覆盖）、恢复原位置、恢复原时值、全部恢复。一切恢复动作等于覆盖了一组新的 Edit Parameter。

# 高阶音高线塑形系统（曲线控制）

## 局部坐标系与随动变换
**绝对不允许曲线游离于音符之外。** 所有绘制的数据点必须映射在以 NoteBlob `StartTime` 为原点、`Duration` 为边界的相对局部坐标系中。
当 NoteBlob 被时间拉伸或平移时，内部手绘线与结构线会自动按比例缩放并随动，始终紧贴载体。

## 自由手绘工具（HandDraw）
直接用类似 Figma 铅笔的工具在 NoteBlob 表面涂抹，按下右键期间，将清理右键扫过的水平区间的绘制。
数据结构表现为与 `original_pitch_curve` 等长的 `NaN` 数组，画过的地方填入实数。在计算 F0 管线时，非 `NaN` 的 HandDraw 点具有最高覆盖优先级。

## 结构化曲线工具（LineControl）

### 曲线形态与锚点控制
类似于 Ableton 的包络线。在音符上拉出结构化的控制线，主要用于处理三类人声微操：
- **颤音（Vibrato）**：表现为波浪线，用户可通过锚点控制波浪的振幅（幅度）、频率（密度），以及两端的淡入淡出。
- **滑音（Glide）**：音符首尾的平滑过渡斜线。
- **转音（Bends）**：带有曲率控制的贝塞尔式折线。

### 影响权重系统（Influence Degree）
每条 LineControl 具备 0% - 100% 的影响力滑块。
管线计算时，`Final F0 = HandDraw (或 Original) * (1 - Influence) + LineControlCurve * Influence`。0 为不起效，100 为当前段彻底被结构线捏造接管。

### 跨块连线展望（未来架构预留）
MVP 阶段的 LineControl 限缩于 NoteBlob 内部。底层结构预留“跨 Blob 引用指向”，为未来研发跨越两个断开音符块的“连音滑行轨”留有余地。

# 界面与交互视觉框架

## 顶部系统导航与走带控制
包含：播放、暂停、停止、循环模式开关。全局时间指针（支持按秒显示，预留按拍显示）、BPM 显示区。

## 主工作区（核心编排场）

### 钢琴窗、网格
左侧垂直显示钢琴键盘键位与八度标识，点击琴键可发声。工作区中央绘制水平时间线网格与垂直音高横格。MVP 支持按秒或固定 BPM 划分竖网格。

### 悬浮式沉浸工具栏
摈弃传统软件顶端塞满按钮的设计。在编辑区底端或鼠标附近悬浮 Figma 式的工具胶囊组件，包含：指针（移动/选定）、拉伸、剪刀、胶水、画笔（HandDraw）、钢笔（LineControl）、响度工具。

## 侧边管理面板

### 左侧：轨道视图层级
在 Standalone 模式下展示所有轨道的列表头（包含名称、Mute、Solo 按钮、音量推子）。支持单轨全屏独占显示或多轨透视。

### 右侧：动态属性与多选管理
类似于 Inspector 面板。无选中时显示项目/轨道级宏观数据；单选时显示该块的具体起止时间、偏移值和恢复按钮。多选时提供批处理接口。

## 全局状态反馈底栏
底部贴边的一条极简进度条，负责映射渲染状态机的红绿脏区。清晰告知用户“当前时间线哪里是可以安全试听的（绿），哪里仍在后台 AI 推理中（红）”。

# I/O 与工作流规范

## 导出引擎规范

### Mixdown 与 Stems 导出逻辑
提供完整的导出对话框。
- **Mixdown**：将工程内未被Mute的轨道汇总混音，导出一个独立的波形文件。
- **Stems**：展示轨道多选框列表（排除已Mute轨道），将勾选的轨道分别独立导出为同等时长的单文件（便于发给混音师）。

### 目标重采样与通道补全
因为内部法定格式为 44.1k Mono，导出时若用户选择 48k/96k 或 Stereo：
- 输出模块在最后写入硬盘前执行重采样升频。
- 若选 Stereo，直接复制 Mono 数据到左右两路。
- 这一切仅发生在 I/O 输出层，**绝不能污染编辑层的原始 44.1k 真相**。MVP 阶段彻底摒弃在工作区内进行真双声道（Stereo 相位差）的复杂显示与编辑支持。

## 持久化与工程落地

### SQLite 单文件工程（Standalone 独占）
Standalone 的定稿落盘格式为单 SQLite 文件，不生成附属的“Audio Folder”（全内联），扩展名为“MLDK”（MeLoDiK）。
- **表结构**：包含 `schema_version`（版本控制门禁，用于未来的前后向兼容）、会话元数据表、轨道结构表。
- **二进制 BLOB**：NoteBlob 私有的 `source_audio_44k`、`original_pitch_curve`、`source_f0_hz`、`source_voiced_probability` 以及编辑参数等必要资产以 BLOB 格式直接灌入数据库。像 `cached_source_mel_log` 这类可重建缓存不落盘。
- **运行时隔离**：软件运行期间的撤销历史栈（Undo/Redo History）属于内存态，绝不允许被序列化存入数据库。工程关闭，历史清空。

### VST 插件态缓存与生命周期
DAW 宿主通过 `getStateInformation / setStateInformation` Chunk 接口来持久化插件状态。
我们必须遵守宿主体积约束的哲学。通过内存管理和高度压缩的序列化树，将当前唯一轨道的参数与必要的音频 BLOB 打包写入 Chunk。宿主工程保存，Melodick 状态随之落盘。

# 命令系统与历史管理

## 命令设计模式
严禁在 UI 事件回调中直接修改 NoteBlob 数据。
一切修改必须封装派生自 `Command` 基类。例如 `TranslateNoteBlobCommand`，内部记录操作前与操作后的真信源快照。

## 撤销重做边界判定
通过统一的 Command Manager 进行入栈管理，确保“每一次鼠标左键按下到松开”成为一个完整的事务（Transaction）边界，保证撤销行为颗粒度符合人类直觉。改变选区不产生历史记录，只有更改“真信源”才入栈。
