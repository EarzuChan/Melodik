# MeloDick 完整设计稿

## 1. 产品定位

MeloDick 是一款面向人声单旋律编辑的修音与再合成软件。

它不是传统意义上的“自动修音插件壳”，也不是只会把音高吸附到音阶上的 Auto-Tune 式工具。它的目标是成为一款以“可视化音符编辑 + 连续音高线编辑”为核心的人声编辑器，方向上接近 Melodyne，但不照搬 Melodyne。

MeloDick 的基本判断如下：

- 目标对象是人声单旋律，不做复音和弦分离，不做 Melodyne 的 polyphonic 模式。
- 编辑的核心是“音符块拥有自己的音高线片段”，而不是一条脱离音符块独立存在的全局编辑曲线。
- 原始分析阶段可以得到整段连续 F0，但进入编辑阶段后，音高线必须被切入各个音符块内部，由音符块持有。
- 最终用于驱动试听、导出和后端处理的完整轨道，是按当前音符块状态实时拼装出来的结果，而不是编辑域中的唯一真相对象。
- HandDraw 不是附属功能，而是 MeloDick 与 Melodyne 拉开差异的关键能力。

一句话描述：MeloDick 要把“人声修音”从参数式工具，升级成“人声音符对象编辑”。

### 1.1 输入边界（GIGO）

MeloDick 默认输入是“可用于修音分析的干声”。

MVP 阶段明确不负责以下前处理能力：

- 去混响
- 降噪
- 人声分离
- 背景噪声与伴奏污染修复

即：`good in, good out；garbage in, garbage out`。

## 2. 核心原则

### 2.1 音符块拥有音高线

在编辑语义上，真相不是一条游离于音符块之外的连续曲线，而是：

- 每个音符块拥有自己对应的原始音高线片段
- 每个音符块拥有自己当前的编辑后音高线片段
- 若多个音符块处于连接状态，它们的边界可以表现为连续，但所有曲线归属仍然落在各自音符块内部

最终送入后端或试听系统的完整轨道，只是在当前时间线上把这些音符块内的内容按规则拼装后的结果。

### 2.2 音符块是主要交互对象

用户最常见的工作方式应该是：

- 看见自动分割出来的音符块
- 选中音符块
- 移动、拉伸、切分、合并、连接、解除连接
- 再在局部用 HandDraw 或锚点精修

而不是面对一条抽象曲线自己从零构建全部结构。

### 2.3 高级编辑必须统一写入同一套数据模型

HandDraw、LineAnchor、音符移动、量化、响度调节、时值调整，都必须落到同一套编辑域模型中，最终共同更新各个 NoteBlob 内部持有的编辑结果，并在渲染前拼装为完整轨道内容。

不能再让不同工具各自维护一套互相打架的状态。

### 2.4 以人声为前提

MeloDick 的音符块不直接等同 MIDI 音符，不是网格内的死矩形，而是“从真实演唱中分析出来的、有原始音高线和时间形态的声乐片段”，这点和 Melodyne 类似。

因此：

- 默认允许滑音
- 默认允许音高波动
- 默认允许不严格对齐到网格
- 默认允许相邻音符存在连接关系
- 默认允许后续重构为更规整的结果

### 2.5 先把编辑语义做对，再做界面拟态

MeloDick 可以借鉴 Melodyne 的交互直觉，但不能只做视觉模仿。核心优先级是：

- 对象模型正确
- 编辑命令正确
- 渲染链闭环正确
- 最后再追求高级 UI 质感

### 2.6 工程状态必须可持久化

MeloDick 不能只做“单次会话编辑器”，必须支持工程保存与恢复。

这不是附加功能，而是基本能力。否则以下链路都无法成立：

- 跨天继续编辑
- 回放/导出前的状态复现
- UI 与渲染缓存协同
- 插件态与宿主工程长期协作

### 2.7 顺序管线是唯一计算模型（已定稿）

每个 NoteBlob 的最终参数计算，固定走以下顺序管线：

- `original_pitch_curve`
- `handdraw_patch_midi`（单实例，数组长度与原始曲线一致；空值点为 `NaN`）
- `line_patches`（多实例，可表达滑音/颤音/自由线）
- `global_transpose_semitones`
- `time_ratio`
- `final_pitch_curve`（供 UI 呈现与重合成输入）

这条管线是唯一真相计算路径，不再并行维护多套“工具私有状态”。

### 2.8 工程落盘定稿（已定稿）

工程文件采用 SQLite 单文件落盘，不再使用文本工程格式。

- 会话、轨道、Blob 元数据为结构化表字段
- 大数组（源波形、原始 F0、patch、mel 等）以二进制 BLOB 存储
- 使用 `schema_version` 做版本门禁与迁移

## 3. 产品最终形态

MeloDick 的最终形态是一款具备以下能力的人声编辑器：

- 导入人声音频
- 自动分析原始音高轨、能量、发声/清音区段
- 自动将人声按“演唱音符/字”的粒度分割为可编辑音符块
- 用钢琴窗视图直观展示这些音符块
- 允许像 Melodyne 一样直接拖拽音符进行音高与时值编辑
- 允许切分、合并、复制、粘贴、连接、解除连接
- 允许独立音符在时间轴上重叠
- 允许对音符块内部持有的音高线做高级编辑
- 允许在音符内和音符间构建滑音、转音、颤音等控制形态
- 使用音符块级的目标音高片段驱动后端再合成人声，并拼装为完整轨道
- 回放、对比、撤销重做、导出结果

## 4. 核心概念

## 4.1 Track

一个工程中可以有多个声轨。每个声轨都拥有独立的 NoteBlob 集合、编辑状态与渲染缓存。

MVP 阶段可以只重点做好单声轨，但数据模型应从一开始就支持多轨。

Track 至少应具备以下混音与监听属性：

- `name`
- `mute`
- `solo`
- `gainDb`
- `orderIndex`
- `blobs`

行为约束：

- 任一轨 `solo=true` 时，仅播放所有 `solo` 轨。
- 无 `solo` 轨时，播放所有 `mute=false` 轨。
- `gainDb` 是轨级增益，作用于该轨最终混音前结果。
- 轨道本身不绑定“源音频文件引用”。
- 导入后的音频会被标准化、分析并肢解为 NoteBlob；轨内只保留 NoteBlob 真相。

## 4.2 Clip

MeloDick 当前工程真相不使用 Clip 作为持久化对象。

导入后的音频在会话中会直接转化为 NoteBlob 资产：

- 每个 NoteBlob 持有该段源波形（44.1k mono float32）
- 每个 NoteBlob 持有该段原始音高曲线与 patch 模型（手绘/线条/移调/时值倍率）
- 每个 NoteBlob 可持有源 mel 等渲染所需中间特征缓存

因此工程恢复不依赖外部音频路径，不存在“工程打开时再去找源文件”的前置条件。

## 4.3 OriginalPitchTrack

原始音高轨，是分析器从输入人声中提取出的连续 F0 轨。

它应保留：

- 时间轴
- F0 值
- voiced / unvoiced 状态
- 置信度
- 可选的平滑结果

它表达的是“原始唱法”的整段分析结果。

但它只属于分析层。进入编辑层后，系统会基于分割结果，把它切成多个归属于 NoteBlob 的原始音高线片段，供恢复和编辑映射使用。

## 4.4 AssembledPitchTrack

AssembledPitchTrack 不是编辑层的主对象，而是按当前 NoteBlob 状态临时拼装出的完整目标轨表示。

它服务于：

- 后端整轨试听
- 导出前的完整轨道准备
- 可视化总览

它由以下内容按时间顺序与连接关系拼装而来：

- NoteBlob 的结构变化
- 每个音符块内部的 `originalPitchSlice`
- 每个音符块内部的 `editedPitchSlice`
- 音符主音高偏移
- 音符时值拉伸
- HandDraw 自由绘制
- LineAnchor 控制点
- 滑音、颤音、量化等高阶修饰

它是渲染前的拼装产物，不应反过来成为 UI 编辑的唯一真相。

## 4.5 新音符块 NoteBlob

“新音符块”不是老项目里那种残缺的 DrawNote 片段，也不是单纯 UI 矩形。它是 MeloDick 的主编辑对象。

一个 NoteBlob 表示一段被系统识别为“可作为一个人声音符对象来编辑”的时域片段。

在新模型里，音符块不是引用一条外部全局曲线，而是直接拥有属于自己的原始片段和编辑片段。

它至少应包含：

- `id`
- `startTime`
- `endTime`
- `originalStartTime`
- `originalEndTime`
- `originalPitchSlice`
- `editedPitchSlice`
- `sourceAudio44k`
- `sourceMelLog`
- `originalPitchSummary`
- `displayPitch`
- `pitchOffset`
- `timeStretch`
- `loudnessGain`
- `linkPrev`
- `linkNext`
- `isDetached`
- `localTimeMapping`
- `analysisTags`

其中：

- `originalPitchSlice` 是该音符块持有的原始音高线片段，用于恢复与参考。
- `editedPitchSlice` 是该音符块当前要送去渲染的目标音高线片段。
- `sourceAudio44k` 是该音符块持有的源波形片段，不依赖外部文件回读。
- `sourceMelLog` 是该音符块可缓存的源 mel 特征，用于降低反复重算成本。
- `displayPitch` 是界面展示的主音高，是从该音符块当前内部曲线派生出来的显示值。
- `pitchOffset` 表示音符整体移调。
- `localTimeMapping` 表示该音符块内部曲线如何映射到当前时间长度。
- `analysisTags` 用于标记这个音符更像稳定音、滑音段、装饰音、长音颤音段等。

术语与代码迁移约定：

- 领域术语统一为 `NoteBlob`。

## 4.6 能力层（Capability Layer）

MeloDick 的核心应按“能力”组织，而不是按“AI模块/分割模块”组织。

能力定义如下：

- `F0ExtractionCapability`：输入音频，输出连续 F0、voiced、置信度。
- `NoteBlobSegmentationCapability`：输入音频 + F0/能量等特征，输出 NoteBlob 列表。
- `VoiceResynthesisCapability`：输入（梅尔/条件特征 + 目标 F0 等），输出重合成音频。

说明：

- “分割”不是 AI 的附属边角，而是独立能力。
- “渲染”也不是 UI 后处理，而是核心能力。
- 这些能力可由 ONNX/本地算法/未来模型实现，但上层编排不应绑定具体实现。

## 4.7 Link

音符之间的“连接关系”是独立概念，不等同于时间相邻。

需要区分三种关系：

- 时间相邻：前后在时间轴上相接
- 编辑连接：编辑一个音符时，邻接音符需联动
- 曲线连续：两音符的音高线在边界处保持连续过渡

用户可手动打断或建立连接关系。

连接状态的意义：

- 默认分割出的相邻音符通常处于连接状态
- 连接状态下，移动和拉伸一个音符会影响相邻音符的时值边界
- 解除连接后，音符可被独立移动
- 独立音符允许重叠

在渲染语义上，还应引入 `RenderGroup`：

- 多个处于连接状态、且时间上连续的音符块，可以组成一个 RenderGroup
- RenderGroup 内部允许作为一次连续渲染任务提交给后端
- 这样既不破坏“曲线归属音符块”的编辑语义，也能减少后端过碎的任务开销

## 4.8 PitchEditLayer

PitchEditLayer 不是一条全局悬浮曲线，而是附着在 NoteBlob 或音符连接边界上的编辑信息层。

它描述的是“这个音符块内部的原始片段如何变成当前编辑片段”，而不是某个单独工具的私有数据。

它可以包含：

- 音符整体偏移
- 音高量化结果
- HandDraw 曲线
- Anchor 控制点
- 段内滑音模板
- 段间转音模板
- 颤音形态
- 局部微调补丁

如果某个修饰跨越两个连接音符存在，它也应显式挂在这段连接关系上，而不是脱离音符块独立存在。

## 4.9 Loudness Editing

响度调节是新音符块未来必须支持的一级能力。

它应基于每个 NoteBlob 的相对响度控制，而不是全局增益旋钮。

未来可以延伸出：

- 音符级响度
- 头尾淡入淡出
- 中段强调点
- syllable 级包络

## 5. 关键编辑语义

## 5.1 选中与主编辑对象

中央编辑区中的主编辑对象是 NoteBlob。

单击选中一个音符块时，右侧属性面板展示其：

- 主音高
- 开始/结束时间
- 时值
- 响度
- 连接状态
- 恢复动作

多选时，右侧切换为批量编辑模式。

## 5.2 音高移动

上下拖拽音符块时，不是只修改一个离散 MIDI 音，而是将该音符块内部的 `editedPitchSlice` 整体按半音进行垂直平移。

结果是：

- 音符主音高显示变化
- 音符内部已有 HandDraw 线条随之整体上移/下移
- Anchor 控制点也随之移动

## 5.3 时值拉伸

左右拖拽音符块边缘时，本质上是改变该段时间映射。

结果是：

- 音符块在时间轴上变长或变短
- 其内部 `editedPitchSlice` 按新时间映射重采样
- 如果该音符与邻接音符保持连接，则会推动邻接边界
- 如果已解除连接，则仅改变该音符自身位置和长度

## 5.4 切分与合并

切分：

- 将一个 NoteBlob 在选定时间点拆成两个新音符块
- 拆分后两边默认继承原曲线对应片段
- 拆分后的连接关系按规则重建

合并：

- 将相邻或逻辑上可合并的多个 NoteBlob 合并为一个更长的新音符块
- 合并后内部曲线保留，不强行抹平

## 5.5 解除连接与重建连接

解除连接后：

- 两侧音符不再互相推动边界
- 可独立移动
- 可形成间隙或重叠

重建连接时需要满足：

- 时间上无其他音符阻塞
- 两音符在时间上可首尾衔接或允许用户指定新边界
- 用户明确执行连接命令

## 5.6 时间重叠

独立音符允许时间重叠。

这不是异常状态，而是高级编辑的一部分。它可用于：

- 制作装饰音
- 制作双层过渡
- 预备未来的交叠处理策略

当时间重叠存在时，渲染层不应让多个音符块去争夺“同一条唯一 F0/频谱定义”。

正确语义应是：

- 每个重叠音符块各自独立生成自己的渲染结果
- 最终在时间线上进行叠加或混音

MVP 阶段可以先采用简单叠加与基础增益管理，不必一开始就做复杂的交叠能量整形。

## 5.7 双击量化到主音高中心

双击音符块时，可执行“把该音符主音高吸附到最合理音级中心”的快捷操作。

这不是把整段曲线强行拉平，而是：

- 先识别该音符块的主导音高中心
- 选择最匹配的半音格
- 计算整体偏移
- 将整段目标音高线整体移到新的中心附近

若用户开启更强的量化模式，可再附加压缩内部偏差的操作。

## 5.8 恢复操作

每个音符块应支持：

- 恢复原音高线
- 恢复原位置
- 恢复原时值
- 恢复响度
- 全部恢复

这些恢复动作应基于 OriginalPitchTrack 被切入该 NoteBlob 时保存下来的原始片段，以及初始分割状态，而不是简单撤销命令。

## 6. HandDraw 与 LineAnchor

## 6.1 HandDraw 的地位

HandDraw 必须保留，并升格为一等能力。

它不是旧系统里“在某段时间上多画一条修正线”的补丁工具，而是直接编辑某个 NoteBlob 内部 `editedPitchSlice` 的自由曲线工具。

HandDraw 的行为应满足：

- 在某个音符块内部绘制时，自动绑定到该音符块
- 拖动音符块时，用户绘制的线随音符块一起移动
- 拉伸音符块时，用户绘制的线按时间映射同步拉伸
- 复制音符块时，绘制线一并复制
- 切分音符块时，绘制线也被切分

## 6.2 LineAnchor 的新职责

LineAnchor 不再作为旧式独立修音模式存在，而是作为“有结构的音高线编辑手法”保留。

它的职责主要是：

- 在音符内部创建有控制感的滑音形态
- 在相邻音符间创建转音或过渡
- 在长音内部塑形颤音轮廓

它和 HandDraw 的关系应当是：

- HandDraw 适合自由绘制
- LineAnchor 适合结构化控制

二者最终都写回所属音符块的 `editedPitchSlice`，必要时再通过连接关系生成跨音符的连续过渡。

## 7. 分割系统设计

## 7.1 问题定义

MeloDick 的分割目标不是“找到所有频率突变点”，而是把人声单旋律切成“适合以音符对象进行编辑”的片段。

这件事不能只靠 F0，也不能只靠能量，需要多特征联合判断。

## 7.2 分割输入

分割器至少读取以下分析结果：

- 原始 F0
- voiced / unvoiced 状态
- 置信度
- 能量包络
- 时间连续性
- 音高变化速度
- 可选的 onset-like 边界特征

## 7.3 分割管线

建议采用分层管线，而不是一开始追求单一大模型：

1. 原始特征平滑
2. 候选边界提取
3. 边界打分与筛选
4. 初步切段
5. 段级修正与合并
6. 生成 NoteBlob 列表

候选边界可能来自：

- pitch jump
- voiced 与 unvoiced 的切换
- 短暂能量谷
- 发音起点
- 持续滑音中的局部稳定点

## 7.4 分割后的人工修正

自动分割不需要神到一步到位，但必须做到“结果可继续编辑”。

因此应提供：

- 手动切分
- 手动合并
- 重新连接
- 解除连接
- 重新定义边界

MVP 阶段的目标不是完美分割，而是“自动结果大体可用，人工修正成本低”。

## 8. 音频与渲染链

最终渲染链应为：

1. 导入音频
2. 提取原始分析结果
3. 自动生成 NoteBlob
4. 将整段 OriginalPitchTrack 切入各 NoteBlob 的 `originalPitchSlice`
5. 用户修改音符块与其内部曲线
6. 根据连接关系与时间连续性生成 RenderGroup
7. 为每个 NoteBlob 或 RenderGroup 生成渲染输入
8. 逐渲染单元调用后端再合成
9. 将渲染结果按时间拼接或叠加
10. 维护时间轴级的渲染缓存与状态
11. 回放与导出

后端渲染层的原则：

- UI 怎么编辑，与渲染层解耦
- 渲染层优先以音符块为单位工作，避免多个重叠音符在单条全局 F0/频谱定义上互相打架
- 对于连接且连续的音符块，渲染层可提升到 RenderGroup 粒度，以减少后端调用与拼接负担
- 重叠音符的语义应是“各自独立发声后叠加”，而不是在同一时刻抢占唯一 F0
- 完整轨道是音符块级渲染结果的拼接或混合产物
- 音频线程不得被昂贵计算阻塞
- 所有重渲染应尽量 chunk 化和增量化

## 8.1 渲染状态条

MeloDick 应提供一条时间轴级的渲染状态条，位置可以在编辑区底部或其他稳定可见的位置。

这条状态条用于直接表达“哪些时间段已经可播放，哪些时间段仍需后台生成”。

建议语义：

- 绿色：该时间段的当前版本已完成渲染，可直接播放
- 红色：该时间段已变脏，当前版本尚未渲染完成
- 可选的进行中色：该时间段正在后台渲染

变脏条件至少包括：

- 音符块位置变化
- 音符块时值变化
- 音符块内部音高线变化
- 连接关系变化
- 影响后端输入的参数变化

## 8.2 懒渲染

渲染策略不应是“每次编辑后立刻整轨重渲染”，而应采用懒渲染：

- 用户修改后，先把受影响时间段标记为红色脏区
- 用户从某个播放点开始播放或预览时，从该播放点向后优先渲染
- 已有绿色缓存的时间段直接复用
- 真正不会被立刻听到的部分可以延后生成

这种方式更接近音频工作站和视频软件的预览逻辑，能显著减少无意义计算等待。

## 8.3 渲染边界与编辑边界

系统内部应明确区分两套边界：

- 编辑边界：NoteBlob
- 渲染边界：RenderGroup 或更细的时间缓存块

这样可以同时满足：

- 编辑体验始终以音符块为主
- 一串连接音符可合并渲染，减轻后端负担
- 重叠独立音符仍能分别渲染后再叠加
- 脏区能按时间范围精确回退为红色并按需补算

## 8.4 工程保存与恢复（Project Persistence）

### 8.4.1 目标

工程保存要保证“可还原编辑真相”，不是只保存导出结果。

下次打开工程时，至少要完整恢复：

- 轨道及其混音状态
- NoteBlob 列表与连接关系
- 每个音符块的 `original_pitch_curve`
- 每个音符块的 patch 模型（`handdraw_patch_midi`、`line_patches`、`global_transpose_semitones`、`time_ratio`）
- 每个音符块的源波形片段与必要中间特征缓存（如 source mel）
- 命令历史（至少可选恢复最近编辑上下文）
- 脏区状态（可选）

### 8.4.2 最小工程格式（MVP）

MVP 工程格式定稿为 SQLite 单文件，语义必须满足“工程自给自足、与外部源文件脱钩”。

最小结构：

- `meta`：`schema_version`、`session_sample_rate`、`duration_seconds`
- `tracks`：轨道混音状态与顺序
- `blobs`：每个 NoteBlob 的元数据 + 二进制 BLOB 资产

Blob 资产最小字段建议：

- `original_pitch_curve`
- `handdraw_patch_midi`
- `line_patches`
- `source_audio_44k`
- `source_mel_log`

### 8.4.3 保存语义

- `Save`：事务写入当前 SQLite 工程文件。
- `Save As`：新建 SQLite 文件并完整写入。
- 自动保存：写入独立 autosave SQLite 文件。

### 8.4.4 不应作为唯一真相保存的内容

以下内容属于可重建缓存，不应成为“唯一真相”：

- 渲染后的整轨临时波形缓存
- GPU/CPU 推理中间张量
- UI 临时选中状态（可选保存，不可依赖）

可以保存“上次播放头位置”等体验状态，但不能影响核心可恢复性。

### 8.4.5 Standalone 与插件态差异

- Standalone：工程文件本身是主真相，不依赖外部源音频路径。
- 插件态：宿主 DAW 工程是一级真相，插件需保存“可序列化的编辑状态块”。

插件态建议保存：

- 音符块结构与编辑曲线
- 分析摘要或可重建索引
- 参数快照

插件态若受宿主序列化体积约束，可采用分级缓存策略，但不得破坏“可恢复编辑真相”。

### 8.4.6 版本迁移

工程文件必须具备向前迁移机制：

- 读取旧 `schema_version` 时执行迁移器
- 迁移失败要给出明确报错，不可静默破坏数据
- 迁移后写回新版本结构

这一步在 MVP 就要预留，不然后期改模型会断档。

## 8.5 脏区语义：派生属性，不是手工标注

“脏”不是要靠外部去打标的状态，而是由编辑真相派生出来的结果。

唯一真相源应是：

- NoteBlob 当前编辑状态（位置、时值、内部曲线、连接关系、响度等）
- 渲染缓存对应的输入快照（revision/hash）

派生规则：

- 若某时间范围内任一 NoteBlob 的当前 revision 与缓存 revision 不一致，该范围即为脏。
- 若一致，则该范围为干净。

这样做能避免：

- 标脏/清脏漏操作导致状态错乱
- UI 状态与渲染状态分叉
- 回放与导出结果不一致

## 8.6 原始音频直通策略（Pass-through）

如果某个 NoteBlob 在当前工程中满足“未编辑”条件，可直接复用原始音频对应片段，不必重渲染。

未编辑判定建议至少包括：

- 音高线等于原始（`editedPitchSlice == originalPitchSlice`）
- 无时值拉伸
- 无响度变化
- 无连接关系变化导致的边界重算

只要任一条件不满足，该 NoteBlob 应进入渲染能力链。

## 8.7 时值拉伸策略

时值拉伸不应采用“渲染后音频机械拉伸”作为主策略，主策略应是重渲染。

推荐结论：

- 只要发生时值拉伸，即使音高线未改，也要重新走 `VoiceResynthesisCapability`。
- 原因是目标时长变化会改变梅尔-时间结构与条件对齐，机械拉伸会引入明显伪影且不稳定。

可选优化（非主链）：

- 允许机械拉伸作为低质量实时预听。
- 后台完成模型重渲染后再无缝替换为高质量结果。

## 8.8 会话音频标准与导入规范

MeloDick 会话标准采样格式定义为：

- `44.1kHz`
- `mono`
- 统一内部时间轴

位深策略修正：

- 会话内部处理与缓存统一使用 `float32`，避免多次编辑和叠加时的量化损失。
- `16-bit` 作为默认导出格式，而不是内部计算格式。

导入规范：

- 输入可为任意常见采样率、声道数和位深。
- 导入时统一转换为会话标准格式（44.1kHz mono float32）。
- 导入完成后，后续分割、编辑、重渲染均基于会话内部素材进行。

这条规范与 HiFiGAN 44.1k 工作点对齐，可显著减少重复的跨采样率往返转换。

## 8.9 导出规范（Mixdown / Stems）

导出必须支持两类目标：

- Mixdown：导出整工程缩混后的单文件。
- Stems：按轨导出，每轨一个文件，可选附带工程导出清单。

导出参数至少包含：

- 采样率：`44.1k`（默认）、`48k`、`source`、`custom`
  注：`source` 在产品语义上指“用户指定采样率”，不再依赖外部源文件元数据。
- 位深：`16-bit`（MVP 默认），后续扩展 `24-bit` 与 `32f`
- 声道：`mono`、`stereo`（mono 导出时直接输出单声道，stereo 导出时通道复制）

实现约束：

- 导出采样率若不等于会话采样率，在导出阶段统一重采样。
- 导出重采样属于 I/O 层能力，不应污染编辑层和渲染层真相。

## 9. 界面最终方案

## 9.1 顶部结构

从上到下：

- 标题栏：系统原生
- 菜单栏：自绘
- 主顶栏：走带控制、当前播放位置、时间显示、调性与自动探测

主顶栏中建议包含：

- 播放、暂停、停止、循环
- 当前时间或播放头位置
- 缩放控制
- 调性显示
- 自动探测调性按钮
- 未来可扩展节拍与小节显示

MVP 阶段可以先按秒显示，不强依赖节拍系统。

## 9.2 主工作区

主工作区为左中右三栏。

### 左栏：轨道面板

负责展示：

- 声轨列表
- 当前高亮编辑轨
- 轨道静音、独奏、显示状态

未来支持的显示模式：

- 仅显示高亮轨
- 高亮轨正常显示，其他轨虚化参考
- 所有轨同等高亮显示

### 中栏：编辑区

这是软件的核心。

它包含：

- 左侧竖直钢琴键盘
- 中央钢琴窗式编辑区域
- 音符块显示
- 原始音高线与目标音高线显示
- 播放头
- 可选网格

交互重点：

- 点击钢琴键可发出对应参考音
- 音符块可拖拽、拉伸、切分、合并
- 手绘线与 Anchor 在同一视图中编辑
- 工具栏浮在编辑区底部，类似 Figma 风格，而不是占据顶部主导航

工具栏建议包含：

- 指针工具（点选和拖拽移动）
- 拉伸工具
- 切分/合并工具
- 连接/解除连接工具
- 音高画笔
- Anchor 工具（画滑音、颤音）
- 响度工具

### 右栏：属性面板

右栏显示当前选中音符块或多选集合的属性。

对于单个音符块，应至少支持：

- 主音高
- 起止时间
- 时值
- 响度
- 与前后音符的连接状态
- 恢复原音高线
- 恢复原位置
- 恢复原时值
- 全部恢复

未来扩展：

- 滑音强度
- 颤音范围
- 头尾淡入淡出
- 中间锚点

## 9.3 收起逻辑

左右栏都可以收起。收起后，在对应角落显示悬浮“展开按钮”胶囊。

这个行为是合理的，但它属于可用性增强，不是第一优先级。

MVP 可以先实现简单折叠，不必一开始就追求很复杂的动画和胶囊设计。

## 10. 命令系统与撤销重做

MeloDick 需要明确的编辑命令模型，而不是 UI 事件直接改状态。

至少应有以下命令：

- MoveNote
- StretchNoteStart
- StretchNoteEnd
- SplitNote
- MergeNotes
- DetachNote
- LinkNotes
- QuantizeNoteCenter
- DrawPitch
- EditAnchors
- ChangeLoudness
- RestorePitch
- RestoreTiming
- RestoreAll

这样做的原因：

- 便于撤销重做
- 便于多选批处理
- 便于未来做宏操作
- 便于保持渲染刷新边界清晰

## 11. 与 Melodyne 的关系

MeloDick 应明确借鉴 Melodyne 的是：

- 音符块式编辑心智
- 直接拖拽式交互
- 音高与时间的可视化编辑方式

MeloDick 应明确不照搬 Melodyne 的是：

- 复音和弦分离路线
- 把音符块内部曲线能力弱化到只剩“音符偏移”
- 过度依赖单一量化式修音

MeloDick 的差异化方向是：

- 专注人声单旋律
- 把音符块内部音高线编辑做强
- 让 HandDraw 成为真正高级、自然、可继承的能力
- 让音符块和曲线之间的映射关系保持透明可控

## 12. 明确不采用的旧方案

以下方案应明确废弃：

- 旧 DrawNote 模式
- 让不同工具分别维护各自的 correction 段
- 把音符当成单纯时间框选结果
- 只允许时间上无重叠的严格序列
- 让 UI 状态直接等于渲染状态

## 13. 最终判断

MeloDick 的终极目标不是“更好看的修音小工具”，而是“以人声音符对象为中心的人声编辑器”。

这要求产品从一开始就建立三层统一认知：

- 原始演唱在分析层可以是连续的
- 编辑对象是拥有自身音高线片段的结构化音符块
- 渲染输出是由音符块级内容拼装出来的完整轨道

只要这三层关系是清楚的，后续无论加响度、滑音、颤音、复制粘贴还是更强的 AI 分割，系统都能继续长大。
